Cam3 Horror Game

The Goal
Cam3 is a personal project built around a core design tension: watching the monster through security cameras keeps you safe, but every second the camera is on it drains your battery. The goal was to create a horror game where the player's only real tool is attention — and it costs something. Beyond the concept, this project became my main sandbox for learning how to build clean, decoupled systems that could each be developed and tested independently.
What I Built
  • Camera system with switching logic and battery drain mechanics
  • Monster state machine with states: Idle, Moving, Stunned, Shocked, Retreating, Lured, Hallway Endpoint, and At Door
  • Event-driven visibility system — not per-frame polling, fires only on state change
  • Modular folder architecture under Assets/Systems/ with a shared Core/ layer
  • Each system designed to be independently sellable and testable
What I Learned
This is the project where decoupled architecture stopped being a theory and became a habit. After a previous project failed because everything depended on everything else, I rebuilt Cam3 from the ground up with strict system boundaries. I now understand why event-driven design outperforms per-frame checks for AI responsiveness — and how to structure a state machine that's actually readable and extendable months later, not just at the time of writing.
