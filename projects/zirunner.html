ZiRunner

The Goal
ZiRunner started as a challenge to build a real-time multiplayer endless runner where multiple players can race simultaneously with smooth, reliable synchronization across all clients. The goal wasn't just to make a game — it was to design systems where data ownership and network authority were deliberately thought through from day one, not patched in later.
What I Built
  • Multiplayer synchronization using Photon PUN with clean authority separation between host and clients
  • Core endless runner gameplay loop with procedural obstacle generation
  • Object pooling system to keep runtime performance stable as the game speeds up
  • Modular data management architecture for player state and score synchronization
  • Scalable systems designed to support variable player counts without code changes
What I Learned
This project pushed me to properly understand the difference between what a client owns and what the server owns. I learned how to design systems around network constraints rather than fighting them. Object pooling went from a concept I knew about to something I now reach for by default in any performance-sensitive context. Most importantly, I saw firsthand how early architectural decisions either create problems or prevent them — and that lesson carries into everything I build now.
