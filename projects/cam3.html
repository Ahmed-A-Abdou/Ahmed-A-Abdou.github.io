Cam3 Horror Game

The Concept
Cam3 is built around a single deliberate constraint: the only tool that keeps you safe costs something to use. Watching the monster through security cameras keeps it at bay — but every second the camera runs drains your battery. Attention itself becomes a resource. Beyond the design concept, Cam3 became my primary architecture project — rebuilt from the ground up after a previous version failed because everything depended on everything else. Every system in this game was designed with strict boundaries and zero coupling between them.
Camera System
Modular security camera management built around a Singleton-managed controller. Players switch between independent camera points, and each camera is a fully self-contained component with no dependencies on other systems. The controller handles switching logic; individual cameras handle nothing beyond their own state.
Visibility Detection
Real-time target visibility combining viewport frustum checks (Camera.WorldToViewportPoint) with raycasting for occlusion testing. Fires state-change events only — no per-frame polling overhead. Detection triggers on camera switches and target movement, keeping the system responsive without burning CPU cycles every frame.
Enemy AI — 7-State FSM
A finite state machine with seven states — Idle, Moving, Stunned, Retreating, Lured, and two threat-escalation states — implemented using the State design pattern with a shared context object. Each state is independently testable and communicates exclusively through a centralized state machine controller. Zero coupling to unrelated systems means adding or modifying a state never touches anything outside its own class.
Resource Management
Battery system with multi-source drain calculation — idle drain, active-tool drain, and instant-cost drains — all configured via ScriptableObjects, fully separated from runtime MonoBehaviour state. Includes probability-based failure triggering for unpredictable difficulty pacing that a designer can tune without touching code.
Secondary Threat System
An independent timer-based escalation system with player counterplay mechanics — timed interaction to de-escalate. Completely decoupled from the primary AI system; neither system knows the other exists.
Architecture — Event-Driven SO System
All inter-system communication runs through a custom ScriptableObject event system using a publisher/subscriber pattern. No direct references exist between systems. This means any system can be developed, tested, or replaced in isolation. Typed events use C# Actions where needed; everything else routes through the SO event layer. The result: zero circular dependencies across the entire project.
Player Controller
Custom FPS controller built on Unity's new Input System and CharacterController, with state-based mode switching between stationary and mobile player states. Input handling is cleanly separated from movement logic.
Engineering Approach
SOLID principles throughout. Single-responsibility per system. ScriptableObject-driven configuration for designer iteration without code changes. Consistent property encapsulation patterns. Every system was built to be independently sellable, testable, and replaceable — because the previous version of this project wasn't, and that's what broke it.
